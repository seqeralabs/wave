Wave exposes an API for dynamic container requests and a registry-compliant service for serving images. Wave runs as a hosted Seqera service or as a self-hosted deployment.

## Requesting and serving images

Wave clients such as Nextflow and the Wave CLI call the Wave API. You can reference an existing container image or supply build instructions — a Dockerfile, Singularity recipe, Conda packages, Conda environment file, PyPI requirements, or Pixi manifest.

When a Wave client makes a request:

- Wave returns a unique URI. Standard container clients pull from it.
- Wave processes the request asynchronously. Wave builds or augments the image as needed.

Wave implements the Docker Registry v2 API and acts as a fully OCI-compliant container registry. Once a container is requested, Wave manages the pull and delivery:

- Wave holds the client connection while builds, augmentations, and scans complete.
- Wave retrieves base layers directly from the source registry. Wave serves added or modified layers itself.
- Containers behave as if they came from a standard registry.
- Wave fits existing workflows and tooling.

Dynamic modification of images lets you customize containers for specific workflows without a full rebuild. You save time and resources.

## Wave image URIs

### Ephemeral containers

By default Wave returns ephemeral containers. These are temporary URIs for single use. They expire after 36 hours. Ephemeral image names use the format:

```
wave.seqera.io/wt/<access-token>/wave/build:<checksum>
```

In the example:

- `<access-token>` is a 12-character unique access token. It acts as a one-time key to the image. It unlocks retrieval using credentials stored in Seqera Platform. No Docker-client authentication is needed.
- `<checksum>` is a 16-character unique build identifier. It ensures correct caching by the Wave backend and by the client. The checksum changes whenever the build recipe or source changes.

### Stable containers

Wave can persist, or freeze, a container build to a registry. Wave then returns the registry URI in place of an ephemeral URI. These URIs are stable. They include no access token. They do not expire. They point at the target registry directly. The client pulls straight from that registry. Stable image names use the format:

```
your.registry.com/library/imagename:<checksum>
```

`<checksum>` is the same 16-character build hash used by ephemeral images. It ties the image to its build inputs. You get reproducible, cache-friendly identifiers.

### Seqera Containers

Seqera Containers is a special case of stable (frozen) containers. It is a free service powered by Wave and run by Seqera and AWS. Anyone can request container images built from Conda or PyPI packages. Wave hosts the results on the community registry for at least five years. Images live at [community.wave.seqera.io](https://community.wave.seqera.io/). See [Seqera Containers](./seqera-containers.mdx).

### Mirroring

Wave can also serve images from private container registries without changing the image name or digest. See [Container mirroring](./mirroring.mdx).
